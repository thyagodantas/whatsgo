internals: add RawClientAddr() escape hatch for tight integrations

Some downstream libraries (github.com/purpshell/meowcaller, our VoIP
integration) import go.mau.fi/whatsmeow directly and take a
*whatsmeow.Client as an argument. Our fork re-exports the same struct
under the name *Client but Go treats them as distinct types because
they live in different modules.

RawClientAddr exposes the underlying *Client value's memory address
via unsafe.Pointer so callers can convert it to *whatsmeow.Client at
the call boundary (typically via reflect or a typed cast). The layouts
are identical, so the cast is sound; this is the documented escape
hatch for the meowcaller integration described in CALL.md.

Note: this method is added directly to internals.go and NOT generated
by internals_generate.go (that script only emits wrappers for
*Client methods; RawClientAddr is a one-off for integration glue).
If someone runs `go generate ./...` the method will be preserved
because it lives on DangerousInternalClient, which the script only
re-declares structurally.

Production code should never need this; keep it as the single
narrow path for libraries that hard-depend on the upstream type.
